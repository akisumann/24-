# 七年巫女神：AI向け引き継ぎ書

この文書は、別のAIへゲームの現状・設計意図・変更時の注意点を渡すためのものです。

## 最優先（絶対ゲート）：物語開始前の必須データ確認

物語開始・再開、または一定年数を進行させた後の描写を行うときは、**必ず最初にゲームが指定する完全セーブデータを直接読み込む**こと。この確認を終えるまで物語本文を出力してはならない。

### 必須手順

1. `STORY_START_*.json` などの開始用概要データがあれば確認する。
2. 概要データの `fullSave` に指定された完全セーブファイルを取得する（本ゲームの完全セーブは `save.js` の `exportSaveGame()` が書き出す `miko-save-year-<年>.json`、または自動保存 `localStorage: mikoGameSave_latest`。中身は `buildSaveData()`）。
3. 完全セーブ内の以下を確認する（かっこ内は本ゲームでの所在）。
   * 経過年（`year`）
   * 神レベル（`godLevel`、神羅記録は `shinraMikos`）
   * 現役巫女全員（`mikos`：50人の完全名簿）
   * 親類縁者・氏族構成（`kin`）
   * 次回大儀候補（`mikos` の20〜34歳から素質上位10人として導出）
   * 現在の時代（`mikos` の役務構成から `MikoEra` で導出。テーマは `role-balance.js` 参照）
   * 時代の出来事（`eventLog`：`events.js` が完全セーブに含める）
   * 出来事が要求した役務（`eventLog[].threat` / 対応role）
   * 役務の適合度（担い手数 `eventLog[].count` と成否 `eventLog[].success`）
   * 出来事の結果と影響（`eventLog[].success` / `delta` / `text`）
   * 直近の進行履歴（`history`）
4. 概要データと完全セーブの年・ルールセット（`ruleset`）が一致することを確認する。
5. 確認した実データだけを使って物語を開始する。

### 開始禁止条件（いずれか該当なら物語を開始しない）

* `fullSave` で指定されたファイルが存在しない。
* 完全セーブが空・不完全・解析不能（`validateSaveData` を通らない）。
* 概要データと完全セーブで年・ルールセットが一致しない。
* 現在の時代、時代の出来事、役務適合結果など、ゲーム画面に存在する情報を確認できていない。
* 現役巫女の完全な名簿を確認できていない。

該当する場合は、確認できたデータと欠けているデータを明示する。古いセーブ、別データ、ソースコード上の一般ルール、AIによる再計算や推測を代用品として使ってはならない。

### 禁止事項

* 完全セーブを読まずに、約300年などの進行結果を文章だけで生成する。
* ソースコードの仕様から、実際に発生した出来事を推測する。
* 過去の完全セーブを現在データとして使用する。
* 人物名、氏族人数、時代、出来事、成功・失敗を創作で補完する。
* 「ゲームデータに見当たらない」と結論する前に、開始概要・完全セーブ・現在読み込まれている拡張ファイルを確認しない。

### 出力前チェック（本文出力の直前に内部確認）

- [ ] 完全セーブを直接読んだ
- [ ] 年とルールセットが一致した
- [ ] 現役巫女の実データを確認した
- [ ] 現在の時代を確認した
- [ ] 時代の出来事を確認した
- [ ] 危機と役務の適合結果を確認した
- [ ] 本文中に推測で補った固有名詞や数値がない

一つでも未確認なら、物語本文を出力してはならない。

### 現行実装での注意（データの実態）

* **シードは存在しない。** 本シミュレーションは `Math.random` による非決定的進行で、乱数シードを持たない（`core.js`）。よって「シード一致」の照合はできない。完全セーブの同一性は `ruleset`＋`year`＋`savedAt`＋`nextId` で判定し、これを唯一の正データとする。同じ年から進めても再現はされないため、必ず**その完全セーブそのもの**を読む。
* **`STORY_START_*.json` は現状ゲームが出力していない。** 書き出しは単一の完全セーブ（上記）のみ。概要ファイルが無い場合は、完全セーブを直接その正データとして扱う。
* `eventLog` は直近 `MAXLOG` 件に丸められる（`events.js`）。それ以前の出来事は完全な形では残らない点に留意する（残っている範囲だけを事実として使う）。

## 最優先：長期シミュレーション後の物語進行

AIがこのゲームを用いて数百年を進行させ、その時点から物語を始める場合は、物語を書く前に必ずゲーム内で実際に生成された現在データを確認すること。

確認対象には、少なくとも現役巫女50人の名前、年齢、現在能力、最大能力、レベル、役務、役職、固有スキル、性格、髪色、身長、体格、スリーサイズ、母親、出自を含める。また、各氏族の現役人数、親類縁者、元巫女、既婚親族、平均素質、勢力関係、神レベル、神羅巫女記録、直近の大儀と世代交代、現在の世界設定も確認する。

物語の登場人物、中心人物、対立、事件、婚姻、政治、氏族関係、戦闘、儀式、日常の立場は、確認した実データから自然に選ぶ。存在しない人物、実際の能力に合わない強者、根拠のない予言・陰謀・危機・制度・対立を、その場の都合だけで追加してはならない。

特に、誰がその時代の最強か、どの氏族が多数派か、どの氏族が少数でも上位者を持つか、誰が次の大儀で母候補になるか、誰が任期終了を控えているかなど、現在データから既に生じている状況を物語の出発点とする。

数百年分のシミュレーション結果は単なる背景ではなく、その物語世界の確定した歴史と現在である。AIはゲーム結果より自作設定を優先せず、人物の性格・技能・身体的特徴・家系・役務・能力差を実際の描写と行動へ反映する。

物語の基本フレームは、プレイヤーである神が七年ごとに顕現し、選ばれた十人の巫女と交わって次代を残しつつ、この七年に何があったかを過去形の報告として聞く「連作の再会」である（`WORLD_SETTING.md` の「神の顕現と大儀の対話」を参照）。七年進行の一回分は「七年眠る → 顕現 → 十人と会う → 七年間の報告を聞く → 語り合う → 次代を残す → また眠る」という一つの物語サイクルとして扱う。

## 最優先：巫女の外見描写ルール（固定）

巫女を物語に登場させるときは、名前と能力だけで済ませず、外見まで含めて描写することを固定ルールとする。

- 髪色・髪型、身長や体格、胸・腰・尻などの身体的特徴、服装を、なるべく明確に描写する。ゲーム内データ（`body` の髪色・身長・バスト/ウエスト/ヒップ、性格、能力）を根拠とし、それと矛盾しない範囲で髪型や服装の具体を補う。
- 特に巫女服については、見た目・着こなし・前垂れの状態・透け感や布の質感・神聖さと実用性を、重要な要素として扱う（`MIKO_CLOTHING_AND_MARRIAGE.md`／`MIKO_FORMAL_ATTIRE.md`／`MIKO_FORMAL_WEAR.md`／`MIKO_TURNOVER_SETTING.md` を参照）。
- 戦闘・会話・見合い・婚姻・日常など、どの場面でも、誰が出てきたか読者がすぐ分かるように外見描写を入れる。
- 強い巫女や印象的な巫女ほど、固有の見た目の特徴を文章の中でしっかり出す。

要するに、巫女は「名前＋能力」だけで登場させず、髪色・身体的特徴・服装まで含めて描く。

## 3. 基本ゲーム

### 時間進行

- 1ターン＝7年。
- 6歳、13歳、20歳、27歳、34歳の五世代が存在する。
- 現在能力の発現率は年齢ごとに20%、40%、60%、80%、100%。
- 年齢上昇では最大能力そのものは変化しない。

### 巫女

- 現役巫女は常時50人。
- ステータスは以下の7項目。
  - HP
  - MP
  - ATK
  - DEF
  - INT
  - SPD
  - DEX
- 各巫女は既存の固有スキルを5個持つ。
- スキルは戦闘専用ではなく、儀礼、治療、制作、調査、移動、護衛なども含む。

### 七年の大儀

- 20歳、27歳、34歳の正式巫女から最大能力平均上位10人を選ぶ。
- 選ばれた巫女は神との娘を一人ずつ産む。
- 34歳の巫女は大儀後に任期終了。
- 空席は神の娘と国家公募採用者で補充し、50人を維持する。

### 神の娘

各ステータスを独立して次の式で決定する。

```text
round(((母親の該当最大能力 + 神レベル) / 2) × 0.50〜1.50)
```

- 神レベルは通常、現役巫女数と同じ50。
- 娘は母親と同じ姓になる。
- 各項目で別々の乱数を使う。

## 4. 一族と親類縁者

- 同じ姓は同じ一族として集計する。
- これは必ずしも近親家族を意味せず、千年単位では遠い支流も同じ氏族名に含まれる。
- 同じ姓なら、はるか昔に共通祖先がいた氏族としてまとめてよい。
- 現役巫女、元巫女、血縁者、配偶者などが増減する。
- 強い巫女が大儀で娘を残し続けると、同姓の現役巫女が増えて大氏族化する。
- 一方で公募出身者の大当たりによって新しい姓が急増し、古い巫女一族が現役ゼロになって断絶することもある。

## 5. 世界設定上の巫女制度

- 巫女は国家と神の契約に基づく特別な身分・資格である。
- 国家が維持費・教育・医療・生活を負担するため、巫女は国家へ実質的な貢献を行う。
- 長期的には、現役50人を中心に、元巫女、親族、技師、医師、官吏、警備、教育担当など数百人規模の国家機関・共同体へ発展する。
- 巫女は辞職可能であり、制度上は禁止されていない。
- ただし辞職によって一族の現役席、大儀選抜の可能性、将来の神の娘、長期的な氏族繁栄が減るため、心理的・社会的な圧力は存在する。
- この圧力は特別な残酷制度というより、重要な会社員、貴族後継者、国家職などが辞める際に生じる責任や引き継ぎの圧力を大きくしたものに近い。
- 貴族社会なので婚姻は現代的な完全自由恋愛ではなく、国家・巫女院・一族・相手家が政治、血統、土地、役職、子供の扱いなどを交渉する。

## 6. 現在のタイマン模擬戦試作

この機能は現在、試作状態であり、ユーザー承認前の固定版ではない。

### 選手選択

- 現役巫女50人から巫女Aと巫女Bを選ぶ。
- 同一人物同士は不可。
- 二人をランダム選択する機能もある。
- 対戦開始後は自動で3ターン進む観戦型。

### 基本判定

各ステータス値をダイスの最大値として使う。

```text
ATK 68 → 1d68
SPD 52 → 1d52
```

各ターン、選ばれた2能力のダイスを一回ずつ振り、合計値を比較する。

### 第1ターン：得意分野

- 各巫女の現在能力上位2項目を使用。

```text
1d上位1位 + 1d上位2位
```

### 第2ターン：対応力

- 第1ターンで使った上位2項目を除く。
- 残り5項目から各自ランダムに2項目を選ぶ。

```text
1dランダム能力1 + 1dランダム能力2
```

### 第3ターン：切り札

- 各自の最高能力を必ず使う。
- 残り6項目からランダムに1項目を選ぶ。

```text
1d最高能力 + 1dランダム能力
```

### 勝敗

- 各ターンの合計が高い側が1勝。
- 3ターン中2勝以上した側が総合勝者。
- 同点時は、同じ能力組み合わせのままそのターンだけ再ロールする。

### スキル補正

- 各巫女が元から持つ5スキルだけを使う。
- 新しい戦闘専用スキルを追加してはいけない。
- そのターンの攻防で自然に使えそうな既存スキルがある場合、ダイス合計を1.1倍する。
- 複数該当しても倍率は重ならず、1.1倍のみ。
- 非戦闘技能を無理に戦闘へ結びつけない。
- 現在のコードでは、戦闘利用可能とみなすスキル名を `battle.js` 内で暫定的に列挙している。
- この列挙は試作であり、ユーザー確認後に調整する。

### 戦闘描写

各ターンの文章は、主に次を使って構成する。

- 選ばれた2能力
- ダイス結果
- 使用可能だったスキル
- 勝敗
- 数値差

例：

```text
攻撃と素早さで勝利
→ 素早く相手を翻弄し、斬撃を通した。

次に防御と知識で敗北
→ 最初はテンポを取ったが、相手に攻め筋を読まれ、守勢を崩された。
```

数値差が小さければ接戦、大きければ圧倒として描く。

## 8. 現在の注意点

- 模擬戦は既存の七年進行や一族計算へ影響を与えない。
- 勝敗や戦績はまだ保存しない。
- 負傷、死亡、報酬、評判変動もまだない。
- スキル利用可否は暫定判定。
- 戦闘描写は能力の組み合わせを元にした短い定型文。
- 現在の戦闘機能はまずタイマンを完成させる段階であり、部隊戦へ勝手に拡張しない。

## 10. v11で加わった表示レイヤーと拡張設計

v11では、固定版の既存関数を極力書き換えず、グローバル関数を**ラップ／上書き**して機能を足す方式で拡張している。新しい機能を足す際もこの作法に合わせること。

### 拡張の作法

- `render` / `renderDetail` / `renderRoles` / `renderRoleMembers` / `assignments` / `buildSaveData` / `restoreSaveData` などのグローバル関数をラップまたは上書きして機能を追加する。
- DOMは既存要素の `innerHTML` を作り替えず、ノード移動（`dashboard.js`）や後付け要素（寸評・年表・物語）で拡張する。`getElementById` で参照される要素のIDは保持する。
- モジュール間は `window.MikoEra` のような小さなAPIで疎結合につなぐ。
- 読み込み順が重要（後のモジュールが前のラップを包む）。`role-balance.js` を最後に読み込む。
- 各モジュールは自分のスタイルを `<style>` 注入で持ち、`index.html` へは `<script>` 参照のみを足す。

### 固定ロジックへ入れた変更

- `core.js`：`reputation()` を動的な `fame` 方式へ変更（`GAME_RULES.md`「国家評判」参照）。`reputationTarget()` / `syncFame()` を追加した。
- `save.js`：`fame` / `fameYear` / `fameShinra` を保存・復元する。
- `trend.js` と `role-balance.js` が `buildSaveData` / `restoreSaveData` をラップし、`trendLog` / `eraLog` を追記する。

### セーブ形式の追加キー

`fame` `fameYear` `fameShinra`（core/save）、`trendLog`（trend.js）、`eraLog`（role-balance.js）、`eventLog`（events.js）。旧セーブは各モジュールが後方互換で補完する。

### `window.MikoEra` API（`role-balance.js` が公開）

- `current()` → `{picks:[domain,...], key}` または `null`
- `theme()` → `"討魔と国土の時代"` のような文字列
- `isInDemand(role)` → その役務（ステータスキー）が今の時代の上位2ドメインに入るか
- `domainKeyOfRole(role)` / `domainDesc(role)`

### 既知の課題・未対応

- **氏族の一強化**：長期進行で1氏族が現役40人近くまで寡占しうる（強い巫女の娘が残る仕様の帰結）。多様性を保つ抑制は未実装。
- 時代の需要は `events.js` の「時代の出来事」で評判へ接続済み。人口・一族への影響は未接続（現状は評判のみ）。
- `chronicle.js` の物語は「現在の状態」からの生成で、巫女ごとの実際の功績履歴（大儀で産んだ娘の数、就任年など）は蓄積していない。
- `advance70.js` はリポジトリに存在するが `index.html` から読み込まれておらず未使用（七十年進行は `batch.js` が担当）。

## 9. 別AIへの依頼テンプレート
